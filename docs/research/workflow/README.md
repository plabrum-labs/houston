# Workflow

**What this category is:** how multi-step, long-running, or human-in-the-loop work gets executed reliably — durable execution engines (Temporal, DBOS, Restate, Hatchet), event-driven job platforms (Inngest, Trigger.dev), business-process/human-task engines (Camunda, Windmill), connector-breadth automation (n8n, Zapier), batch scheduling (Airflow), and process mining (Celonis).
**Why it's in this research:** any Houston-built app that models a business process needs an execution substrate underneath its actions — this folder maps what "reliable multi-step execution," "human approval as a first-class primitive," and "flow control" already look like across a mature, competitive market.
**Files:** 12.

Temporal is the deep reference implementation for cluster-based durable execution (full-history replay); Restate and DBOS are the two sharpest architectural alternatives (step-checkpoint, not full replay — one as a self-contained binary, one as a library in your own Postgres); Hatchet occupies similar ground to DBOS (Postgres-backed) but as a full orchestration engine, not a library. Inngest and Trigger.dev are narrower, event/job-shaped durable platforms rather than general workflow engines. Camunda is the deep reference for the human-task/business-rule half of the category — a different problem (who approves what, under what rule) than the durable-execution half (did the code finish). Windmill covers similar human-in-the-loop ground more narrowly. Airflow is the batch-scheduling reference, a genuinely different problem (when should this run) from durable execution (how do I not lose progress). Zapier and n8n are connector-breadth plays, not execution-model plays. Celonis is filed here despite being a different category (process mining, not execution) because it operates on the same underlying primitive every workflow/BPM engine in this folder produces or could produce — an event log of (case, activity, timestamp) — and answers the adjacent question "did the process defined here actually execute as designed," which is the analytics layer on top of everything else in this folder.

## Convergence

**A queue's contract ends at delivery; durable execution's contract is completion — and multiple vendors converge on exactly this framing, not just the same underlying mechanism.** Inngest states it directly: "What happens inside the job, whether it completes, and what to do if it fails midway are outside the queue's contract." Restate cites the same framing directly and builds its whole pitch around it. The practical heuristic, stated in `restate.md`: work that's atomic, self-contained, describable as a single verb (send, process, sync) fits a queue; work that's "process-shaped" — multi-step, long-running, needs "and then"/"if that"/"when that" to describe — needs a durable engine, because a mid-execution crash must resume, not restart. Temporal's own case study (Retool, `temporal.md`) is real-world evidence of teams discovering this the hard way: Retool built a hand-rolled queue on PG Boss + Lambda, hit scaling/maintenance limits, and adopted Temporal — then went straight to Temporal for their *second* product rather than repeat the build.

**The versioning/determinism tax is real and independently confirmed across every full-history-replay engine, with the tradeoff explicitly named.** Temporal's core constraint — code must replay deterministically against recorded event history, and even a harmless-looking reorder of two independent `await` calls can break every in-flight instance — is described in detail in `temporal.md` as "viral on user code" and "invisible until an instance actually replays through the changed code path." DBOS and Hatchet converge on the same diagnosis from the opposite side: DBOS's file states plainly that Temporal "replays the entire workflow's control flow... on every recovery" while DBOS instead "memoizes named steps," so a code change that doesn't alter which steps run or their order is safe. Hatchet and Restate independently adopt the same step-checkpoint model DBOS uses. This is a genuine, three-way-converged architectural fork: **full-history replay (Temporal) trades whole-program determinism for a battle-tested, mature ecosystem; step-checkpointing (DBOS, Restate, Hatchet) trades that maturity for forgiving-of-refactors recovery.**

**Human-in-the-loop is converging on the same three primitives, expressed differently per vendor: assign-to-group (not individual) as the default, an approval as a data-collecting step (not a binary gate), and a mandatory timeout on any suspended wait.**
- **Assign to groups, not individuals**: Camunda states this as explicit best-practice guidance — "individual-assignment-first designs die on the first vacation" — via a three-parameter model (`assignee`/`candidateUsers`/`candidateGroups`) settable simultaneously.
- **Approval as structured data, not approve/reject**: Windmill's "Suspend & Approval" renders an arbitrary schema form on the approval page, with submitted values flowing directly into later steps via `resume["..."]` — functionally the same idea as Camunda's DMN-table-driven task assignment, where "who approves this" is itself a business rule with multiple typed outputs (assignee, candidateUsers, candidateGroups) rather than hardcoded logic.
- **Mandatory timeout on suspension is a convergent, explicitly-warned-about default**: Windmill's docs state plainly that without a configured timeout, an under-approved flow "stays suspended indefinitely" — the same failure mode Trigger.dev's waitpoint tokens solve by defaulting to a 10-minute timeout, and the same "don't poll, suspend at zero cost" ethos Inngest's `step.waitForEvent()` and Temporal's wait primitives share. Three unrelated vendors (Windmill, Trigger.dev, Temporal/Inngest) independently treat "suspend cheaply, but never suspend indefinitely by accident" as a load-bearing default.

**Postgres-as-durability-store is a real, multi-vendor architectural bet, not one outlier's idiosyncrasy.** DBOS (library, checkpoints piggyback on the step's own Postgres transaction) and Hatchet (full orchestration engine, Postgres for both runtime *and* observability) independently make the same two claims: `SELECT ... FOR UPDATE SKIP LOCKED` makes Postgres a legitimate high-throughput queue without a separate broker, and SQL-queryable workflow state means dashboards/ad hoc debugging/alerting work for free against ordinary tables — explicitly contrasted against Temporal's opposite choice (Cassandra + Elasticsearch as two separate systems). DBOS reports 30,000+ workflows/second after `SKIP LOCKED` + partial-index optimization (vendor-reported); Hatchet's founding claim citing the same mechanism is explicitly flagged as a founder-stated thesis, not an independent benchmark — the *mechanism* convergence (two vendors, same technique, same justification) is the reliable finding; the specific throughput numbers are not.

**Celonis is a different problem, but it operates on the same substrate this entire folder produces.** The minimal input contract for process mining — (Case ID, Activity, Timestamp) — is exactly what a durable-execution or BPM engine's event history already is, or could trivially be made to emit. Conformance checking ("as defined" vs. "as is") turns "did the process I designed actually run the way I designed it" into a computable diff once that log exists. This is why Celonis is filed in `workflow/` rather than a pure analytics folder: it's the retrospective-analysis layer that becomes available for free to anyone in this category who already has an event log in the right shape — and Celonis's own moat, per its file, is not the mining algorithm (well-understood, academically established) but the hard 80% of reconstructing a usable event log from hostile ERP schemas that were never designed to answer "what happened, in what order, when."

## Worth stealing

- **Signals / Queries / Updates as three distinct message shapes** — async-write / free-sync-read-even-on-completed-workflows / sync-tracked-write, rather than one generic "send event" primitive with polling bolted on to recover the other two behaviors (`temporal.md`).
- **Virtual Objects' per-key intrinsic lock** — serialized-per-key execution as a general substitute for a distributed lock service, with no separate lock-acquisition/release code path to get wrong (`restate.md`).
- **Exactly-once steps by piggybacking the checkpoint on the step's own DB transaction** — a structural guarantee possible specifically because the durability store and the step's own writes are the same Postgres instance (`dbos.md`).
- **`RunChild` as the primary composition primitive** — any I/O or external-system call wrapped as a child task, keeping the parent durable task's body free of non-replay-safe side effects (`hatchet.md`).
- **Five distinctly-named flow-control primitives** (concurrency, throttling — excess queued, rate limiting — excess skipped, debounce, priority, singleton, batching) instead of one overloaded "rate limit" knob (`inngest.md`).
- **Waitpoint tokens, HTTP-completable with no SDK required on the calling side** — a bare POST from a form, a Slack button, or a webhook resumes a suspended, unbilled run (`trigger-dev.md`).
- **The claim-step in the task lifecycle** (create → claim → complete) — the specific mechanism that prevents two people working the same task simultaneously in a shared pool (`camunda.md`).
- **DMN hit policies as real semantic choices, not decoration** — Unique (asserts mutual exclusivity, catches violations), Any (requires agreement), Collect (gathers all matches) vs. First (silently discards overlap-detection) (`camunda.md`).
- **Four-eyes principle enforced as a data condition inside the engine** — an explicit "does not equal" gateway check comparing approver identities, not a UI hint or policy document (`camunda.md`).
- **Data interval as the run's identity, not wall-clock "now"** — what makes backfill a well-defined, idempotent operation rather than an ad hoc rerun (`airflow.md`).
- **Catchup defaulting to false** — Airflow's own history (default-true catchup causing thundering-herd reprocessing after downtime) is the cautionary case for why "auto-run everything I missed" should be explicit opt-in (`airflow.md`).
- **The (Case ID, Activity, Timestamp) event log as a minimal, general-purpose schema** — any system already emitting or adaptable to emit this tuple gets process-mining-style conformance analysis essentially for free (`celonis.md`).
- **A great generic HTTP step as the connector-treadmill escape hatch** — n8n's HTTP Request node and Zapier's explicit, named `Retrieve Poll` fallback both treat "no bespoke connector exists yet" as a supported first-class path, not a degraded or hidden mode (`n8n.md`, `zapier.md`).

## Worth avoiding

- **Determinism is not opt-in and not locally scoped** — a Temporal workflow-code change invisible in a diff (reordering two independent `await` calls) can break every in-flight instance on replay; the mitigation (`GetVersion`/Build-ID versioning) is itself an ongoing maintenance surface, and getting the marker ordering wrong reproduces the exact failure it exists to prevent (`temporal.md`).
- **DBOS's "exactly-once" is narrower than the marketing language suggests** — it covers steps whose side effects are Postgres writes through DBOS's own transaction API in the same instance as the checkpoints; a step calling an external API still falls back to at-least-once with idempotency left to the caller (`dbos.md`).
- **Camunda's Tasklist V2 authorization regression** — as of 8.8 (V2 default), candidate-user/group restrictions are no longer evaluated for task visibility; authorization collapsed to process-definition level, a documented, shipped capability gap rather than a hypothetical risk. The transferable lesson: fine-grained authorization has to be a hard invariant from the start, or a rewrite silently drops it while the coarse check still returns *something* (`camunda.md`).
- **Airflow's repeated, specific history of timezone/DST scheduling bugs** — non-trivial cron hour ranges computing the wrong first execution after a DST transition, a scheduler crash triggered by a DST boundary, DAGs in `Europe/Brussels` stopping entirely around the October transition. The remediation (timezone-aware datetimes everywhere, explicit DST-boundary testing) needs to be centralized in the scheduling primitive itself, not left to every workflow author (`airflow.md`).
- **The connector ecosystem is not a winnable battleground to compete on directly** — Zapier's ~8,000–9,000 connectors represent continuously-maintained integration surface a challenger cannot realistically match head-on; the documented alternative is a great generic HTTP step plus a small number of truly enterprise-critical connectors, not breadth-first parity (`zapier.md`).

## Gaps

- **No engine in this set documents how its flow-control primitives compose when stacked** — Inngest's file notes this explicitly (a singleton function that's also throttled): individually simple, but combination behavior isn't documented anywhere in the source material.
- **Mid-flight reconfiguration of a human-approval gate is unevenly supported and thinly documented.** Camunda explicitly supports modifying due date/priority/assignment on running instances; Windmill's file notes the source material didn't surface whether required-approval count or approver list can change mid-flight.
- **The versioning tax has no vendor-agnostic solution** — every full-history-replay engine pays it, every step-checkpoint engine avoids it at the cost of ecosystem maturity; nobody in this set has a documented way to get both.

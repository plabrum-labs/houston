# Cryo — v1

Cryo is the control plane that hosts Houston's vertical app backends across a scaling pool
of machines. Each backend is a self-contained static Go binary that sits idle the large
majority of the time. Cryo starts each backend on demand when a request arrives, places it
on a machine with capacity, proxies traffic to it, and stops it once it goes idle. The machine
pool grows and shrinks with the actively-used set, so cost tracks the backends awake at any
moment rather than the size of the roster.

Active backends run as processes on some machine in the pool; idle backends are not running
anywhere. A request for a cold backend places and starts it; a stretch of silence stops it and
frees its machine capacity.

## Per-machine control plane

A control-plane process runs on every machine. It owns the machine's exposed port and is the
only component the load balancer health-checks — individual backends start and stop without
affecting machine health.

**Routing.** The control plane maps each request to a backend by Host header. If the backend
runs on this machine, it proxies straight to that backend's unix socket with sub-millisecond
overhead. If the backend runs elsewhere, the control plane resolves its machine from the
placement registry and forwards the connection there. If the backend is not placed anywhere, it
asks the scheduler to place it.

**On-demand start.** A request for a backend not yet running on this machine triggers a start:
the control plane takes a per-backend start lock so concurrent first requests cause a single
start, fetches the binary if it isn't cached locally, spawns the backend process (assigning it a
unix socket and an isolation context), waits for readiness, then proxies the held request.

**Process supervision.** The control plane tracks each running backend's state, process, and
socket, and waits on exit. Unexpected exits restart with exponential backoff; a crash-looping
backend trips a circuit breaker that returns errors quickly rather than restarting in a tight
loop.

**Idle reaping.** The control plane tracks each local backend's last-seen time and stops
backends idle past a threshold — SIGTERM, then SIGKILL — draining in-flight requests first,
freeing memory, and clearing the backend's placement entry. A backend marked pinned-warm is
never reaped.

**Isolation.** Each backend runs as a host process in its own cgroup v2 (memory limit, CPU
weight) under its own uid/gid, confined by seccomp and dropped capabilities.

**On-demand binaries.** Binaries live in S3. A machine fetches a backend's binary on first
start, verifies its checksum and signature, caches it on local disk, and LRU-evicts cold
binaries — a machine holds only the binaries for the backends it runs, not the whole roster.
Deploying a backend pushes a new binary to S3 and updates its manifest version; a machine picks
up the new version on the next start and drains-and-swaps a running backend.

**Capacity reporting.** Each machine reports its free capacity (spare memory) and health to the
fleet scheduler.

**Observability.** The control plane captures each backend's stdout/stderr tagged per backend
and forwards it, and exposes per-backend and machine-level metrics.

## Fleet control

A single elected leader owns placement and scaling decisions, so machines never race on shared
state.

**Placement registry.** A shared, consistent store (Redis) holds `backend → current machine`.
Machines read it to route non-local requests; the scheduler writes it on placement and reaping.

**Scheduling.** When a request wakes a backend that is not currently placed, the leader selects
a machine with free capacity using best-fit packing — filling machines before spreading —
records the placement, and the chosen machine starts the backend. If no machine has capacity,
the leader triggers a scale-up and places the backend on the new machine.

**Membership.** The leader tracks live machines through service discovery. When a machine
fails, its placement entries are invalidated and its backends re-place on other machines on
their next request.

**Autoscaling.** The leader scales the machine pool on fleet memory utilization: adding machines
ahead of saturation — kept responsive by a warm pool of pre-booted instances — and removing
them when utilization falls, draining a machine's backends through a termination lifecycle hook
before it goes away. The pool holds a floor of two machines across availability zones. Scale-up
is aggressive and scale-down is conservative, so the pool does not thrash a placement reshuffle
on every fluctuation.

**Consolidation.** As backends start and stop, machines fragment. The leader migrates backends
off sparse machines onto fuller ones in the background — draining a running backend so it
restarts on its target — to reclaim emptied machines. Migration is gentle and infrequent, since
moving a running backend costs it a restart.

## Runtime

**Backend contract.** A backend is a static Go binary that accepts connections on the unix
socket the control plane provides, signals readiness once, and serves requests. Backends hold
no durable in-process state: sessions live in Redis and application data in Postgres, so a
backend can be stopped, restarted, or moved to another machine transparently. A backend opens
its database pool lazily on first use and releases it on exit, so a stopped backend holds no
connections. Startup does no warm-up work.

**Background work.** Scheduled and queued jobs run in a separate always-on worker fleet backed
by Redis, independent of the on-demand backends, so a stopped backend never blocks job
execution.

## Boundaries

The control plane fetches and runs backends, places them across machines, and proxies traffic
to them. It does not build binaries (that is the build pipeline's job), does not own
application data or database access, does not perform application-level authentication, does not
provision infrastructure (it runs on infrastructure provisioned for it, reading its fleet, ALB,
storage, and data-plane endpoints from the launchpad layer's stack outputs), and does not manage
east-west traffic between backends. It is an ingress proxy, a process supervisor, and a
placement scheduler, nothing more.

## Cost profile

**Per-backend footprint.** An idle backend consumes nothing. A running backend holds ~15–25 MB
RSS and near-zero CPU between requests.

**Fixed overhead:** a floor of two machines across availability zones, one shared Aurora
Serverless v2 Postgres, one Redis (ElastiCache), and one ALB.

**Fleet compute** packs backends onto EC2 Graviton. A roster of a thousand backends with a small
active fraction fits on the two-machine floor; the pool grows only as the concurrently-active
set grows. At a thousand backends with a small active fraction, compute is on the order of
~$0.15–0.25 per backend per month.

**Scale-to-zero economics.** Steady-state compute tracks the *concurrently active* subset, not
the total roster; total count drives cached-binary disk usage and cold-start frequency. Adding
mostly-idle backends leaves steady-state compute flat — the pool grows only when more backends
are awake at once.

**Provisioning.** The pool is sized for *peak* concurrency, not average: a burst that wakes many
backends at once must fit in machine memory, and machines are added ahead of saturation to
absorb it. A backend that receives sparse but recurring traffic is kept pinned-warm rather than
paying a cold start on every wake.

**Capacity.** Each machine holds hundreds of running backends; the pool scales to hold the
active set; the roster is effectively unbounded, since a dormant backend costs only a registry
entry and, where cached, the disk its binary occupies.

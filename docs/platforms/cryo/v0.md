# Cryo — v0

Cryo is the control plane that hosts Houston's vertical app backends on a single machine.
Each backend is a self-contained static Go binary that sits idle the large majority of the
time. Cryo starts each backend on demand when a request arrives and stops it once it goes
idle, so only the actively-used backends consume memory while the rest stay cold.

Every backend's binary is present on the machine's local disk. Active backends run as
processes; idle backends are not running at all. A request for a cold backend starts it; a
stretch of silence stops it.

## Control plane

A single control-plane process runs on the machine. It owns the machine's exposed port and is
the only component the load balancer health-checks — individual backends start and stop
without affecting machine health.

**Routing.** The control plane is the request entry point. It maps each request to a backend by
Host header and proxies the connection to that backend's unix socket. A running backend is
proxied straight through with sub-millisecond overhead.

**On-demand start.** A request for a backend that isn't running triggers a start: the control
plane takes a per-backend start lock so concurrent first requests cause a single start, spawns
the backend process (assigning it a unix socket and an isolation context), waits for the
backend to signal readiness, then proxies the held request. With the binary already on local
disk, a start is a process fork — tens of milliseconds.

**Process supervision.** The control plane tracks each running backend's state, process, and
socket, and waits on exit. A backend that exits unexpectedly restarts with exponential backoff;
a crash-looping backend trips a circuit breaker that returns errors quickly rather than
restarting in a tight loop.

**Idle reaping.** Because the control plane sees every request, it tracks each backend's
last-seen time and stops backends idle past a threshold — SIGTERM, then SIGKILL — draining
in-flight requests first and freeing the backend's memory. A backend marked pinned-warm is
never reaped, keeping latency-sensitive backends hot.

**Isolation.** Each backend runs as a host process in its own cgroup v2 (memory limit, CPU
weight) under its own uid/gid, confined by seccomp and dropped capabilities, so no backend can
exhaust the machine or read another backend's files.

**Observability.** The control plane captures each backend's stdout/stderr, tags it per
backend, and forwards it, and it exposes per-backend and machine-level metrics that host
container metrics cannot see.

## Runtime

**Backend contract.** A backend is a static Go binary that accepts connections on the unix
socket the control plane provides, signals readiness once, and serves requests. Backends hold
no durable in-process state: sessions live in Redis and application data in Postgres, so any
backend can be stopped and restarted transparently. A backend opens its database pool lazily on
first use and releases it on exit, so a stopped backend holds no connections. Startup does no
warm-up work.

**Binaries.** Every backend's binary lives on the machine's local disk. Deploying a backend
replaces its binary; the next start runs the new version, and a running backend is drained and
swapped.

**Background work.** Scheduled and queued jobs run in a separate always-on worker process
backed by Redis, independent of the on-demand backends, so a stopped backend never blocks job
execution.

## Boundaries

The control plane fetches and runs backends and proxies traffic to them. It does not build
binaries (that is the build pipeline's job), does not own application data or database access,
does not perform application-level authentication, does not provision infrastructure (it runs
on a machine provisioned for it), and does not manage east-west traffic between backends. It is
an ingress proxy and a process supervisor, nothing more.

## Cost profile

**Per-backend footprint.** An idle backend consumes nothing — it isn't running. A running
backend holds ~15–25 MB RSS and near-zero CPU between requests. The machine's memory bounds how
many backends run at once.

**Fixed overhead:** one machine (EC2 Graviton), one shared Aurora Serverless v2 Postgres, one
Redis (ElastiCache), and one ALB.

**Scale-to-zero economics.** Memory is consumed only by the concurrently-active backends; the
idle roster costs only the disk its binaries occupy. A machine hosting a hundred backends of
which five are active runs those five — roughly 125 MB — and nothing for the other ninety-five.

**Capacity.** A single machine holds hundreds of running backends (memory-bound at ~15–25 MB
each) and a roster limited only by the disk its binaries occupy. Cryo v0 serves the fleet
as long as the concurrently-active set fits one machine's memory and the roster fits its disk.

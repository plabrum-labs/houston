# Data — v0

The data layer is Houston's shared data plane: the logical model layered on the one Aurora
Postgres cluster that Houston's `infra/` provisions. It does not own the physical cluster —
`infra/` provisions and runs that — it owns everything logical inside it: the schemas, the
database roles and their grants, the row-level-security policies that isolate every tenant and
every customer org, and the migration mechanism that evolves each schema. Its job is to let
hundreds of independent vertical apps share one Postgres instance while each app sees only its own
world, holds only least-privilege access, and evolves its own schema without ever touching a
neighbor's.

## One cluster, many schemas

Every app shares a single Aurora Postgres cluster. Separation is by **schema**, not by database or
by instance. The cluster holds one **platform schema** owned by Houston, and one **app schema** per
vertical app. Tenant, app, schema, and the app's database role are all one-to-one: `arive` the
company is `arive` the app is the `arive` schema is the `arive_runtime` role. An app's schema is its
entire world in the database — its tables, its indexes, its migration history — named from the app's
identity so no two apps' objects collide.

Two isolation axes run through this model, and they live in different places:

- **Tenant** separates one app builder's world from another — `arive`'s data from `marlin`'s. For an
  app's own tables this is **structural**: a different schema behind a different role, needing no
  policy. For the shared platform tables it is enforced by row-level security, described below.
- **Org** separates one customer organization from another *within* a single tenant's app. This is
  row-level security on an `organization_id` column, and it is the mechanism most tables ever need.

## Platform schema

The platform schema holds the identity data shared across the whole fleet — `users` and
`organizations` — and is owned and migrated **only by Houston**. No app migrates it and no app writes
to it directly: an app's `_runtime` role holds `SELECT` on the platform schema and nothing more, and
any change to platform data goes through a Houston API call rather than a cross-schema write. That
keeps a single place enforcing the invariants on shared identity no matter how many apps read it.

Apps read platform tables directly — an app joins its own rows to `platform.users` and
`platform.organizations` freely — so the platform schema carries tenant-level RLS to keep those joins
honest. Every platform table denormalizes a `tenant_id`, and its policy admits only the current
tenant's rows:

```sql
USING (tenant_id = platform.current_tenant())
```

`platform.current_tenant()` is a `STABLE SECURITY DEFINER` function that maps the connected role to
its tenant. It reads `current_user` — the role the app authenticated as — and returns that role's one
`tenant_id`. Because it is `SECURITY DEFINER`, the `_runtime` role needs no grant on the underlying
role→tenant mapping: an app can ask "what is *my* tenant," never enumerate everyone's. Because
`current_user` is fixed for the life of the connection and cannot be changed by application code, the
tenant boundary cannot be forged from inside an app. Postgres folds the function to a single
evaluation per query, so the policy costs one indexed `tenant_id` equality per row.

On platform tables, tenant RLS is the mandatory floor; narrowing to a single org is the app's own
`WHERE` clause, not an enforced policy, because a tenant legitimately runs cross-org views over all
of its customers.

## App schemas and roles

Each app schema carries exactly two database roles, and an app is only ever connected through one of
them at a time:

- **`_migrator`** holds DDL rights on that app's schema **alone** — never the platform schema, never
  another app's schema. It is used solely at deploy time, by the migration mechanism, and never by
  running application code.
- **`_runtime`** holds DML rights (`SELECT`/`INSERT`/`UPDATE`/`DELETE`) on the app's own schema,
  `SELECT`-only on the platform schema, and **no DDL rights at all**. It is the role the running
  backend connects under for every request, and it is also the app's tenant identity: it is the
  `current_user` that `platform.current_tenant()` resolves.

The split is a hard guarantee, not a convention. Schema changes happen only at deploy time, under
`_migrator`, scoped to one schema; a running backend physically cannot alter a table, cannot reach
another app's schema, and cannot write platform data, because the role it holds has no grant that
would let it. And because the app authenticates *as* its tenant role rather than selecting a tenant
after connecting, no bug or compromise in application code can move the tenant boundary.

## Org isolation

Within an app's schema, many customer orgs share the same tables, and row-level security keeps each
org to its own rows. Every org-scoped table carries an `organization_id` column and a policy that
admits a row only when it belongs to the current request's org:

```sql
USING (organization_id = current_setting('app.organization_id')::uuid)
```

The org is carried in a **transaction-local** session variable that the running backend sets at the
start of each request's transaction, from the authenticated user. The check is a single indexed
equality — no joins, no function calls — enforced by Postgres beneath the query. A handler that
forgets to filter by org still returns only the current org's rows, because the database applies the
policy, not the application.

Org is app-set rather than role-bound because it is a boundary *inside* the tenant the app already
owns — the app is trusted to say which of its own customers a request belongs to, exactly as far as
that tenant's own data and no further.

## Operator-private tables

A tenant may hold data that belongs to none of its customers and must be invisible to all of them —
proprietary pricing models, internal cost formulas, margin tables that drive the product but are not
customer-facing. Such a table uses the same machinery with a different discriminator, admitting rows
only for an operator-scoped session:

```sql
USING (current_setting('app.actor_type') = 'operator')
```

Any customer-scoped request (`actor_type = 'customer'`) gets zero rows from these tables regardless
of what the app's Go code does — even a handler that accidentally joins the proprietary table into a
customer query returns nothing from it. The tenant's own operator-context work sets
`actor_type = 'operator'` and sees it.

## Every table has an explicit posture

No table is left without an RLS policy. Each one declares, in its migration, how it is scoped —
org-isolated, operator-private, or another deliberate posture — so a table's visibility is never an
accident of which grants happen to exist. A table with no policy at all is a defect the migration
mechanism surfaces, not a valid state.

## The session-scope contract

The policies above read their inputs from two sources, and this layer owns both:

- **Tenant** comes from `current_user` — the role the backend authenticated as — resolved through
  `platform.current_tenant()`. The app never sets it; it is who the app is.
- **Org and actor type** come from transaction-local session variables — `app.organization_id`,
  `app.actor_type`, `app.actor_id` — that the running backend sets with `SET LOCAL` at the opening of
  each request's transaction, before any query runs. Because they are transaction-scoped they reset
  at commit and never leak from one request into the next on a reused connection.

This layer defines the variable names, their types, and the guarantee that every scoped policy reads
them; the running backend honors the contract by setting them from the authenticated identity on
every request.

## Migrations

An app's entire migration responsibility is a `migrations/` folder in its repo — schema-scoped
migration files and nothing else. The app declares no roles, holds no credentials, and never runs its
own migrations. This layer owns the mechanism that turns that folder into applied schema:

- **Schema-as-code with `ent` + Atlas.** An app defines its data model once as `ent` schema in Go.
  Atlas diffs that model against the current database state and generates the versioned migration
  files. The model is the source of truth; the migration files are generated from it.
- **RLS policy is part of the schema.** A table's row-level-security policies — its org isolation, its
  operator-private posture — are expressed as code alongside the table definition and versioned into
  the same migration files. A table and its access posture ship and migrate together; there is no
  out-of-band step that grants access separately from the schema that defines it.
- **Per-schema version tracking.** Each app's migration-version history lives inside its own schema,
  so two apps' migration heads never share a table and never collide.
- **Applied under `_migrator`, scoped to one schema.** Migrations run under the app's `_migrator` role
  against the app's schema alone; the mechanism cannot touch the platform schema or another app's
  schema, because the role it runs under cannot.

## Seam with Launchpad

This layer owns the migration mechanism; Launchpad, Houston's deploy platform, invokes it. At deploy
time Launchpad takes the app's `migrations/` folder and applies it against the app's schema, under the
app's `_migrator` role, before the new backend takes traffic — then the new backend connects under
`_runtime` and serves. This layer defines everything about *what* is applied and *how* it is scoped:
the migration file format, the `ent` + Atlas toolchain that produces it, the `_migrator`/`_runtime`
role model and their grants, and how RLS-policy-as-code is expressed. Launchpad is the caller that
runs it at the right moment in a deploy; it defines none of the mechanism.

## Cluster capacity posture

The cluster scales to zero when idle and wakes on the next connection, so an idle fleet incurs no
standing compute cost. Once apps carry steady traffic the cluster stays warm as a matter of course —
ambient requests across the fleet keep it up — and it settles back to zero only during genuine
silence. The logical model is indifferent to this: schemas, roles, RLS, and the session-scope
contract behave the same whether the cluster beneath them is warm or asleep.

## Boundaries

The data layer owns the logical model and nothing physical. It does not provision or run the Aurora
cluster — that is `infra/`, and it already exists; this layer defines what lives inside it. It does
not deploy apps or decide when migrations run — Launchpad applies the mechanism this layer owns, at
deploy time. It does not set the request's identity or open the request's transaction — the running
backend does that, honoring the session-scope contract this layer defines. It is the layer that makes
one shared Postgres safe to divide among hundreds of apps: schema-isolated by tenant, role-least-
privileged, org-scoped by construction, and migrated one schema at a time — nothing above that,
nothing below it.
